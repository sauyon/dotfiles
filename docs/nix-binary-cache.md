# Consuming the self-hosted attic binary cache

These dotfiles configure every machine to pull from the self-hosted **attic**
binary cache on the kube cluster (`https://attic.ko.ag/kube`, cache name
`kube`). Build/server side lives in the `kube` repo (`docs/nix-binary-cache.md`
there); this file is the *consumer* side only.

## Key facts

- **The cache is private.** Anonymous `GET .../nix-cache-info` returns attic's
  own `401` (not a Cloudflare challenge). Pulling therefore needs a **read-only
  token** presented via netrc — there is no anonymous read.
- **Public signing key** (safe to commit, not a secret):
  `kube:YLRejBKnIVKqvZRXBvFR4KmosPZPg9phiM+pRlhbQ+c=`
- **These boxes are Arch + Determinate Nix, multi-user, with `trusted-users =
  root`.** This is the crux: nix **ignores** `substituters` /
  `trusted-public-keys` / `netrc-file` set in a *non-trusted* user's
  `~/.config/nix/nix.conf`. So the cache config must reach the **daemon**, i.e.
  live under `/etc`, not in home-manager. That is why this is wired through
  `system/deploy` (sudo → `/etc`) rather than `home.nix`.

## How it is wired (Linux: utsuho, setsuna, fujiwara)

Three pieces, all applied by `./system/deploy` (sudo):

1. `system/etc/nix/nix.custom.conf` → `/etc/nix/nix.custom.conf`
   Determinate `!include`s this from `/etc/nix/nix.conf`. Adds the substituter
   and trusted key with the **`extra-`** forms (append, don't replace — keeps
   `cache.nixos.org` and FlakeHub).

2. `system/etc/determinate/config.json` → `/etc/determinate/config.json`
   ```json
   { "authentication": { "additionalNetrcSources": ["/etc/determinate/netrc.custom"] } }
   ```
   This is the Determinate-supported hook for extra netrc entries. Determinate
   **merges** the referenced file into its synthesized `/nix/var/determinate/netrc`.

   > Gotcha: do **not** append to `/nix/var/determinate/netrc` directly — that
   > file is regenerated by `determinate-nixd` (e.g. on boot) and your line is
   > lost. `additionalNetrcSources` survives regeneration. And you cannot
   > override `netrc-file` via `nix.custom.conf`: Determinate sets it *after*
   > the `!include`, so the determinate value always wins.

3. `/etc/determinate/netrc.custom` (root, `0600`) — **rendered at deploy time**,
   never committed. `system/deploy` decrypts `atticPullToken` from `secrets.yaml`
   (GCP KMS, as the user) and `sudo install`s `machine attic.ko.ag password
   <token>` to `/etc`. Then it restarts `nix-daemon`.

To apply on a machine: `./system/deploy` (needs your sudo password + your
`~/.config/sops/gcp-key.json` for the KMS decrypt).

## How it is wired (darwin: mari)

`mari` runs nix-darwin-managed nix (`nix.enable = true` in `flake.nix`), so the
config goes straight into `nix.settings` (→ `/etc/nix/nix.conf`) and the netrc is
rendered as root by a `sops.templates` entry. Applied by `darwin-rebuild switch`.
If `mari` is ever switched to Determinate Nix, move this to the `/etc/determinate`
approach above instead.

## The read token

A dedicated **pull-only** token (scope `--pull kube`, no push — so a dotfiles
machine cannot poison the cache). It is stored sops-encrypted as `atticPullToken`
in `secrets.yaml`. To re-mint (from a machine with `kubectl` to the `ko-local`
context):

```sh
kubectl --context ko-local -n attic exec deploy/attic -- \
  atticadm -f /etc/atticd/server.toml make-token \
  --sub dotfiles-pull --validity 10y --pull kube
```

Then replace `atticPullToken` in `secrets.yaml` (decrypt → edit → re-encrypt) and
re-run `./system/deploy` (Linux) / `darwin-rebuild switch` (mari).

## Verify

```sh
nix config show | grep attic.ko.ag                 # substituter is active
nix store info --store https://attic.ko.ag/kube    # prints Store URL = auth OK (was 401)
```
